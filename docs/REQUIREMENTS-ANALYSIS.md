## 2. 요구사항별 상세 분석

---

### R1. 차시별 자산 누적 + 매 차시 월급 지급

**요구사항:**
- 현재: 시나리오 재선택 시 모든 학생 자산이 100,000원으로 리셋됨
- 요구: 차시(세션)가 바뀌어도 이전 차시의 최종 자산이 다음 차시 시작 자산으로 이어져야 함
- 추가: 매 차시 시작 시 월급 100,000원을 지급하여, 돈을 모두 잃은 학생도 계속 참여 가능

**현재 코드 분석:**

| 파일 | 관련 로직 | 문제점 |
|------|----------|--------|
| `playerStore.ts` → `resetAllPlayers()` | `cash = startingCash`, `holdings = {}` 으로 완전 초기화 | 누적 불가 |
| `ScenarioSelector.tsx` | 시나리오 선택 시 `resetAllPlayers()` 호출 | 리셋 트리거 |
| `GameConfig` 타입 | `session: number` 존재 | 세션 번호만 있고 세션 간 데이터 연결 없음 |

**필요한 변경:**

```
1. [DB] sessionResults/ 경로 활성화
   - 차시 종료 시 각 플레이어의 최종 자산/보유종목 스냅샷 저장
   - 구조: sessionResults/{session}/players/{playerId}/{cash, holdings, totalAssets}

2. [Store] playerStore에 새 메서드 추가
   - continueFromLastSession(): 이전 세션 데이터 로드 → 현재 상태로 복원
   - addSalary(amount): 전체 플레이어에 월급 일괄 지급

3. [Admin] ScenarioSelector 또는 새 컴포넌트
   - "차시 이어서 시작" vs "초기화 후 시작" 선택 옵션
   - 월급 지급 버튼 또는 차시 시작 시 자동 지급 설정

4. [Store] stockStore
   - 차시 간 종목 가격도 이어갈 것인지 / 리셋할 것인지 정책 결정 필요
```

**주의사항:**
- 차시가 이어질 때 종목 가격은 리셋하되, 학생 자산(현금 + 보유 주식의 현금 환산)만 이어가는 것이 자연스러움
- 보유 종목을 그대로 가져가면 새 시나리오의 종목 ID가 달라질 수 있으므로, **현금으로 환산 후 이월**하는 것이 안전
- 월급은 차시 시작 시 자동 지급이 운영상 편리 (관리자가 잊을 수 있음)

### R2. 마지막 2차시 금액 리셋 (경연대회 모드)

**요구사항:**
- 9~10차시에서는 누적 자산과 관계없이 동일 조건(예: 100,000원)에서 시작하는 경연대회

**현재 코드 분석:**
- `resetAllPlayers(startingCash)`가 이미 존재하므로, R1의 "이어서 시작" 옵션과 병행하면 됨

**필요한 변경:**
- R1과 함께 구현 시, Admin UI에서 "경연대회 모드 (전원 동일 자금으로 시작)" 체크박스 추가
- 별도의 대규모 변경 불필요

**주의사항:**
- 경연대회 모드에서는 순위판을 반드시 최종 라운드까지 비공개하면 교육적 효과 극대화
- GameConfig에 `isCompetitionMode: boolean` 플래그 추가 고려

### R3. 정보 거래소 시스템 (A/B/C 등급 정보 구매)

**요구사항:**
- 런닝맨처럼 학생이 정보를 "사고 파는" 시스템
- 6개 거래소 (= 종목별? 또는 라운드별?), 각 거래소에 A/B/C 등급 정보
- C등급: 누구나 아는 정보 (저렴), B등급: 괜찮은 정보 (중간), A등급: 핵심 정보 (비쌈)
- 정보 구매 시 개인 현금 자동 차감
- 라운드당 뉴스 2개 + 구매 가능 정보 (C:3개, B:2개, A:1개)

**현재 코드 분석:**

| 항목 | 현재 상태 | Gap |
|------|----------|-----|
| 힌트 시스템 | 3단계 뉴스 (속보/추가보도/긴급속보) 관리자 공개 | 구매 개념 없음 |
| 학생 현금 | `playerStore.updatePlayerCash()` 존재 | 차감 로직은 있음 |
| 정보 데이터 | `ScenarioRound`에 hint1/2/3만 존재 | 등급별 정보 구조 없음 |
| 학생 UI | HintPanel에 공개된 힌트만 표시 | 구매 UI 없음 |

**필요한 변경 (대규모):**

```
1. [Type] 새 타입 정의
   - IntelInfo { id, grade: 'A'|'B'|'C', stockId, content, cost, round }
   - PlayerPurchase { playerId, intelId, purchasedAt }

2. [Data] 시나리오 데이터 확장
   - ScenarioRound에 intels: IntelInfo[] 추가
   - 라운드별 6개 정보 (C:3, B:2, A:1), 각각 특정 종목과 연결
   - 가격 설정: C=1,000원, B=3,000원, A=5,000원 (기획 확정)

3. [DB] Firebase 경로 추가
   - intelPurchases/{purchaseId} — 구매 기록
   - 또는 players/{id}/purchasedIntels: string[] — 플레이어별 구매 목록

4. [Store] 새 스토어 또는 orderStore 확장
   - purchaseIntel(playerId, intelId): 현금 차감 + 구매 기록
   - getPlayerIntels(playerId): 해당 플레이어가 구매한 정보 목록

5. [UI] 새 컴포넌트
   - IntelMarket.tsx (정보 거래소 메인)
   - IntelCard.tsx (등급별 정보 카드, 구매 전/후 상태)
   - StudentView에 새 탭 추가 또는 HintPanel 내 거래소 버튼

6. [Admin] 라운드별 정보 입력 UI (R5와 연결)
```

**설계 고려사항:**

| 고려사항 | 선택지 | 권장 |
|---------|--------|------|
| 거래소 구조 | 종목별 6개 vs 라운드별 1개 통합 | **라운드별 1개 통합** (종목별이면 UI 복잡, 학생 혼란) |
| 정보 공유 | 구매한 정보를 다른 학생에게 알려줄 수 있나? | **시스템으로는 불가**, 오프라인 공유는 허용 (교육적 토론) |
| 정보 중복 구매 | 같은 정보 재구매 가능? | **불가** (이미 구매한 정보는 비활성화) |
| 힌트와의 관계 | 기존 3단계 힌트와 공존? | **분리** — 힌트는 전체 공개 뉴스, 정보는 개인 유료 |
| 정보 열람 시점 | 구매 즉시? 다음 라운드? | **구매 즉시** (정보의 가치가 시간에 따라 감소) |

**잠재적 문제:**
- 현금으로 정보를 사면 투자 가능 금액이 줄어듦 → 정보 비용과 투자 수익 사이의 **밸런스 조정** 필요
- A등급 정보가 너무 노골적이면 교육 효과 감소 → 정보 품질 설계가 핵심
- 시나리오 데이터량이 크게 증가 (라운드당 3+6=9개 텍스트) → 관리 부담 증가, R5(CRUD)의 중요성 상승

### R4. 개인별 종목 손익 퍼센테이지 표시

**요구사항:**
- 학생이 보유한 각 종목에 대해 매수 평균가 대비 현재가의 손익률(%)을 표시
- 관리자도 학생별 손익 현황을 확인하여 수업 운영에 활용

**현재 코드 분석:**

| 파일 | 현재 상태 | Gap |
|------|----------|-----|
| `Player.holdings` | `{ stockId: quantity }` — 수량만 저장 | **매수 평균가(cost basis) 없음** |
| `Portfolio.tsx` | 보유수량 x 현재가 = 평가액 표시 | 손익률 계산 불가 |
| `Leaderboard.tsx` | 총자산 기준 수익률만 표시 | 종목별 손익 없음 |
| `orderStore.ts` | 주문 기록에 price 있음 | 평균가 계산은 가능하나 현재 미구현 |

**필요한 변경:**

```
1. [Type] Player.holdings 구조 변경
   변경 전: holdings: Record<string, number>           // { "ai": 5 }
   변경 후: holdings: Record<string, HoldingDetail>     // { "ai": { quantity: 5, avgPrice: 10200 } }

   interface HoldingDetail {
     quantity: number;
     avgPrice: number;    // 가중 평균 매수가
     totalCost: number;   // 총 투자 원금
   }

2. [Store] orderStore.submitOrder() 수정
   - 매수 시: avgPrice = (기존totalCost + 신규매수금액) / (기존수량 + 신규수량)
   - 매도 시: avgPrice 유지, totalCost 비례 감소

3. [UI] Portfolio.tsx 수정
   - 각 종목: "손익 +12.5%" 또는 "-3.2%" 표시
   - 색상: 수익=빨강, 손실=파랑 (한국 주식 컨벤션)

4. [Admin] StudentStatusList.tsx
   - 학생별 주요 보유종목 + 손익률 요약 추가 (선택)
```

**주의사항:**
- **holdings 구조 변경은 breaking change** — Firebase에 저장된 기존 데이터와 호환되지 않음
- 마이그레이션 전략 필요: 기존 `number` 형태 → `HoldingDetail` 객체로 변환
- `playerStore`의 모든 holdings 참조 코드 수정 필요 (약 10+ 위치)
- 매도 시 FIFO vs 평균법 결정 필요 → **평균법이 단순하고 적합**

### R5. 시나리오 CRUD 관리 폼 (Admin)

**요구사항:**
- 관리자(광희)가 직접 시나리오를 생성/수정/삭제할 수 있는 폼
- 라운드별 뉴스, 힌트, 종목 변동률을 입력
- 리스트로 관리 가능

**현재 코드 분석:**

| 항목 | 현재 상태 | Gap |
|------|----------|-----|
| 시나리오 데이터 | `src/data/scenarios.ts`에 하드코딩 | DB 기반 아님 |
| ScenarioSelector | 하드코딩된 시나리오 목록 표시 | CRUD UI 없음 |
| 시나리오 구조 | Scenario → ScenarioRound[] | 적절, 확장 가능 |

**필요한 변경:**

```
1. [DB] Firebase에 시나리오 저장 경로 추가
   - scenarios/{scenarioId}/info: { name, description, stockCount }
   - scenarios/{scenarioId}/rounds/{roundNum}: ScenarioRound 데이터

2. [Store] 새 scenarioStore 생성
   - CRUD 메서드: createScenario, updateScenario, deleteScenario
   - 라운드 관리: addRound, updateRound, deleteRound

3. [UI] Admin에 시나리오 관리 섹션
   - ScenarioManager.tsx (시나리오 목록 + 생성/삭제)
   - ScenarioEditor.tsx (라운드별 편집 폼)
   - RoundEditor.tsx (뉴스, 힌트 3개, 종목별 변동률 입력)

4. [Data] 기존 하드코딩 시나리오 → DB 마이그레이션
   - scenarios.ts의 데이터를 Firebase에 seed하는 스크립트
   - 또는 최초 실행 시 자동 마이그레이션
```

**설계 고려사항:**
- 종목 변동률 입력 UI: 슬라이더(-30% ~ +30%) vs 직접 입력 → **직접 입력 + 프리뷰가 실용적**
- R3(정보 거래소)과 연계 시 라운드 편집 폼이 복잡해짐 → **탭 분리** (기본 정보 | 힌트 | 정보 거래소)
- 시나리오 복제 기능 유용 (기존 시나리오 기반으로 변형)
- 입력 실수 방지를 위한 **미리보기** 기능 중요

**잠재적 문제:**
- 종목 변동률 합계가 비현실적이면 (예: 모든 종목 +30%) 게임 밸런스 붕괴
- 검증 로직 필요: 변동률 범위 제한, 필수 필드 체크
- Firebase DB 규칙 업데이트 필요 (시나리오 읽기/쓰기 권한)

### R6. 라운드별 회고 메모 + 최종 기록 제출

**요구사항:**
- 각 라운드 종료 시 학생이 "잘한 점 / 아쉬운 점 / 다짐"을 기록
- 최종 라운드 종료 시 모든 라운드 메모를 리스트로 보여주고, 최종 종합 기록 작성 제출
- 이 메모들을 통해 학생의 분석 능력/이해도를 파악 (대학 입시 활용)

**현재 코드 분석:**
- 메모/회고 관련 기능 전혀 없음
- reviewing 단계에서 뉴스 분석(analysis)만 표시됨
- 학생은 결과를 보기만 하고, 입력할 수 있는 인터랙션 없음

**필요한 변경:**

```
1. [Type] 새 타입 정의
   interface RoundReflection {
     playerId: string;
     round: number;
     session: number;
     goodPoints: string;      // 잘한 점
     improvements: string;    // 아쉬운 점
     resolution: string;      // 다짐
     timestamp: number;
   }

   interface FinalReflection {
     playerId: string;
     session: number;
     strategy: string;        // 나의 투자 전략 요약
     learnings: string;       // 배운 점
     finalThoughts: string;   // 최종 소감
     timestamp: number;
   }

2. [DB] Firebase 경로
   - reflections/{session}/{playerId}/{round}: RoundReflection
   - finalReflections/{session}/{playerId}: FinalReflection

3. [UI] 새 컴포넌트
   - RoundReflectionForm.tsx
     → reviewing 단계에서 분석 아래에 표시
     → 3개 텍스트 영역 (잘한 점 / 아쉬운 점 / 다짐)
     → 제출 후 수정 가능

   - FinalReflectionView.tsx
     → ended 단계에서 표시
     → 모든 라운드 메모 리스트 (읽기 전용)
     → 최종 종합 기록 작성 폼

   - StudentView에 reviewing 단계 UI 확장

4. [Admin] 회고 열람 기능
   - 학생별 메모 열람 (리스트)
   - 내보내기(CSV/PDF) 기능 → 입시 서류 활용
```

**설계 고려사항:**
- 메모 작성은 **reviewing 단계에서만** 가능하게 할지, **다음 라운드 시작 전까지** 가능하게 할지
  → **reviewing + waiting 단계 모두 허용** 추천 (학생이 충분히 생각할 시간)
- 메모 길이 제한: 200자 이내 (짧은 회고) vs 무제한 → **200~500자 권장, 제한 없음**

- 입시 활용이 목적이므로 **오타/수정 기능 필수**

### R7. 원/달러 구별 + 환율 시스템

**요구사항:**
- 원화 종목과 달러 종목 분리
- 학생 총 재산을 원/달러로 구별 표시
- 현재 환율 기준 총액 자동 계산 (원화로)

**현재 코드 분석:**

| 항목 | 현재 상태 | Gap |
|------|----------|-----|
| Stock 타입 | sector 필드만 존재 | currency 필드 없음 |
| 자산 계산 | `cash + Σ(qty × price)` 단일 통화 | 다중 통화 미지원 |
| format.ts | `formatCurrency()` → "원" 고정 | 달러 표기 없음 |

**필요한 변경:**

```
1. [Type] Stock에 currency: 'KRW' | 'USD' 추가
2. [Type] Player에 cashKRW, cashUSD 분리 또는 cash: Record<Currency, number>
3. [Type] GameConfig에 exchangeRate: number 추가
4. [Store] 환율 관리 로직 (관리자가 라운드별로 설정 또는 시나리오에 포함)
5. [Store] 자산 계산 공식 변경:
   totalAssets = cashKRW + (cashUSD × exchangeRate)
                + Σ(KRW종목 qty × price)
                + Σ(USD종목 qty × price × exchangeRate)
6. [UI] 종목 리스트에 통화 표기, 포트폴리오에 원/달러 분리 표시
7. [Admin] 환율 설정 컨트롤 (수동 or 시나리오 기반)
```

**잠재적 문제:**
- **복잡도가 급격히 증가**: 주문, 자산 계산, 포트폴리오, 순위판 등 거의 모든 컴포넌트 수정
- 환율 변동 이벤트는 시나리오에 어떻게 녹일지 설계 필요
- 달러로 주식을 사려면 원→달러 환전 기능도 필요한지 결정
- 학생 인지 부하 증가 → **7~8차시 이후에만 활성화** (커리큘럼과 일치)
- R11(Feature Flag)과 연계하여 차시별로 잠금/해제하는 것이 적합

### R12. 금액 매트릭스 (시드/투자/프로세스)

**요구사항 (민영 메모):**
- 시드 금액, 정보 구매 비용, 투자 수익률 사이의 밸런스를 정량적으로 검증

**분석:**
- 이것은 코드 변경보다는 **게임 디자인 밸런싱** 문제
- 핵심 변수: 시작 자금(100,000원), 정보 가격(1,000/3,000/5,000원), 종목 변동률(±5~20%)
- 정보를 모두 구매하면 비용 = C(3x1,000) + B(2x3,000) + A(1x5,000) = 14,000원 (시드의 14%)
- A등급 정보로 확실한 +15% 종목에 투자하면 = 86,000원 × 1.15 = 98,900원 → 정보비 포함 적자

**밸런스 제안:**
```
시드 100,000원 기준 적정 정보 가격 범위:
- C등급: 500 ~ 1,000원 (시드의 0.5~1%)
- B등급: 2,000 ~ 3,000원 (시드의 2~3%)
- A등급: 3,000 ~ 5,000원 (시드의 3~5%)

정보 전략이 유효하려면:
- A등급 정보가 가리키는 종목의 변동률 > 정보 비용 비율
- 예: A등급 5,000원 투자 → 해당 종목 +10% 이상이어야 수익
```
- GameConfig에 정보 가격을 설정 가능하게 하면 차시별 밸런스 조정 가능

## 5. 리스크 및 주의사항

### 5.1 기술적 리스크

| 리스크 | 영향도 | 대응 방안 |
|--------|-------|----------|
| holdings 구조 변경 시 기존 데이터 손상 | 높음 | 마이그레이션 스크립트 + 백업 후 적용 |

| 고려사항 | 제안 |
|---------|------|
| 1~3차시는 기본 기능만 사용 | R11 Feature Flag로 불필요한 기능 숨기기 |
| 4차시부터 정보 거래소 도입 | 첫 도입 차시에는 C등급만 해금, 이후 B, A 순차 해금 |
| 광희가 시나리오를 직접 만들 수 있어야 | R5 우선 구현, 편집 UX 최대한 단순하게 |
| 학생 회고 데이터의 입시 활용 | R6 구현 시 내보내기 형식을 학교 서류 양식에 맞추기 |

## 6. 확장 방향성

### 6.1 단기 확장 (Phase 2 - 4~6차시 대응)
- 정보 거래소 + 시나리오 CRUD + Feature Flag
- 이 세 가지가 확보되면 4~6차시 커리큘럼 운영 가능

### 6.2 중기 확장 (Phase 3 - 7~10차시 대응)
- 환율 시스템 + 종목 확장 + 공매도/ETF
- Admin 페이지 리팩토링 필요 (현재 단일 페이지 → 탭/라우트 분리)

### 6.3 장기 확장 (수업 외)
- **다른 학교/교사에게 제공**: 멀티 테넌트 구조 (학교별 독립 DB)
- **학생 분석 대시보드**: 투자 성향 분석 (공격적/보수적), 정보 활용도, 의사결정 패턴
- **AI 시나리오 생성**: 뉴스 + 힌트 + 변동률을 LLM으로 자동 생성
- **실시간 주가 연동**: 교육 목적으로 실제 주가 데이터 일부 활용 (지연 데이터)
